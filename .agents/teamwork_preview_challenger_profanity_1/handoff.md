# Handoff Report — Profanity Filter Verification

## 1. Observation

- **Implementation Details**:
  - The entry point for the profanity filter is `src/utils/profanityFilter.js`.
  - It loads `leo-profanity` and registers two dictionaries: the default English dictionary and a custom Turkish word list `src/utils/tr-profanity.json`.
  - The custom Turkish dictionary contains 21 words (e.g., `"salak"`, `"siktir"`, `"göt"`).
- **Core Library Behavior** (`node_modules/leo-profanity/src/index.js`):
  - Line 95: `sanitize: function (str) { return str.toLowerCase().replace(/\.|,/g, ' '); }`

    const sanitizedStr = this.sanitize(str);

## 2. Logic Chain

1. **Space-delimited exact match**:
   - `leo-profanity` splits strings using `[^ ]+` (words separated by spaces) after replacing `.` and `,` with spaces.
   - It performs a Set lookup (`this._wordsSet.has(word)`) which requires an exact string match.
   - Therefore, embedded substrings (e.g., `"tamam"` containing `"am"`, or `"assume"` containing `"ass"`) will *not* trigger the filter because the Set lookup is exact, preventing substring-based false positives.
2. **Punctuation Bypass**:
   - Because only `.` and `,` are replaced with spaces in `sanitize()`, other punctuation marks (e.g., `!`, `?`, `-`, `_`) remain part of the word token.
   - For example, `"siktir?"` yields the token `"siktir?"`. Since the dictionary contains `"siktir"` but not `"siktir?"`, the lookup fails, bypassing the filter.
3. **Accent Bypass**:
   - No accent normalization/diacritic removal is done (e.g. `String.prototype.normalize`).
   - Thus, `"şalāk"` (which contains `ş` and `ā`) does not match `"salak"` or `"şerefsiz"` exactly, bypassing the filter.
4. **Turkish Capitalization Bypass**:
   - Standard JavaScript `.toLowerCase()` converts uppercase Turkish dotted `İ` to `'i\u0307'` (combining dot character), which does not match normal `'i'` (`\u0069`).
   - Hence, `"SİKTİR"` does not match `"siktir"` exactly.
   - Similarly, `"SıkTır"` becomes `"sıktır"`, which does not match `"siktir"`. Both bypass the filter.
5. **Profile Title Leak**:
   - `updateProfile(name, title)` only performs the check on the `name` parameter. The `title` parameter is saved without check.

---

## 3. Caveats

- We did not execute live tests on the system since the environment timed out on command permission prompts. However, the logic is verified directly from the source code of `leo-profanity` (v1.9.0) and `src/utils/profanityFilter.js`.
- The exact character representation of `İ` in the underlying JS engine depends on the V8 version, but in all standard implementations it decomposes into combining marks under standard `toLowerCase()`, resulting in a bypass unless normalized.

---

## 4. Conclusion

The profanity filter implementation is robust against false positives on clean names and embedded substrings, but is highly vulnerable to trivial bypasses using punctuation, accents, Turkish case differences, and leetspeak. Furthermore, the `title` field in profile updates is completely unchecked.
